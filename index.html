Setting up a new project

Copy this repository's CLAUDE.md into a new project folder and work through these steps in order.

  1. Create the project folder

    mkdir -p ~/Projects/<project-name>
cd ~/Projects/<project-name>

  2. Specialize CLAUDE.md

    Copy the template in and fill in every bracketed placeholder:

    cp ~/Projects/ProjectTemplate/CLAUDE.md ./CLAUDE.md

    Open it and work through each section:

    • §1 Role and Persona — the agent's specialty and how it should relate to you.
    • §2 Project Context — the central question, goal, or deliverable.
    • §3 Primary References — anchor sources, priority order.
    • §4 Domain Expertise — subdomains the agent can assume without re-explaining.
    • §5 Cross-Cutting Synthesis — delete if the project is narrow; keep and fill in if requests will regularly span multiple sources or subdomains.
    • §6–§8 (Sourcing, Derivations, Interaction Style) — generic and already filled in; adjust only if the project needs a different stance.
    • §9 Formatting — domain-specific conventions. Delete the LaTeX Input Conventions subsection entirely if there's no LaTeX or custom-macro material.
    • §10 Boundaries — add hard, project-specific constraints beyond the generic "never fabricate" rule.
    • §11–§12 (Repository Hygiene, Session Log Artifact) — generic, leave as-is.

  3. Add project content

    Bring in whatever source material the project needs. If a subdirectory is already its own git repository — synced with an external tool such as Overleaf — decide how to handle it before running git init at the project root:

    • Exclude it via .gitignore — simplest; the nested repo stays wholly separate.
    • Fold it in as plain files — delete its .git and let the outer repo track the files directly, losing the sync link.
    • Git submodule — git submodule add <url> <path>, preserving the sync link at the cost of operational overhead.

    Skipping this decision and running a blind git add -A will silently record the nested repo as a broken submodule reference with no URL — a common and confusing mistake.

  4. Write .gitignore

    At minimum:

    # macOS
.DS_Store

# anything decided to exclude in step 3, e.g.:
# some-nested-repo/

  5. Initialize git and make the first commit

    git init
git add CLAUDE.md .gitignore   # plus whatever else belongs in the first commit
git status                     # verify nothing unwanted is staged
git commit -m "Initial commit: project instructions and gitignore"

  6. Create the GitHub repository and push

    Requires the gh CLI, authenticated (gh auth status to check).

    gh repo create <repo-name> --private --source=. --remote=origin --push
# use --public instead of --private if the project has no reason to stay private

    This creates the remote repo, wires it up as origin, and pushes main in one step. Confirm:

    git remote -v
gh repo view <owner>/<repo-name> --json visibility,url,defaultBranchRef

  7. GitHub Pages for a browsable docs site (optional)

    Only do this if the project's content is fine being public. On a personal (non-organization) GitHub account, Pages sites are publicly viewable regardless of whether the source repo is private — private Pages requires a GitHub Enterprise Cloud organization. If the repo is private specifically to keep its content unpublished, skip this step.

    Build a gh-pages branch — commonly as an orphan branch, or via a separate git worktree to avoid disturbing the main checkout — push it, then enable Pages and set the repo's homepage:

    gh api -X POST repos/<owner>/<repo-name>/pages \
  -f source[branch]=gh-pages -f source[path]=/
gh repo edit <owner>/<repo-name> --homepage "https://<owner>.github.io/<repo-name>/"

    This page is that site.

  8. First conversation with Claude Code

    Open the project in Claude Code. The agent picks up CLAUDE.md automatically. On the first substantive turn that produces a derivation, code, or other technical content worth keeping, it starts the running session-log artifact and creates session-log.html per §12 — nothing further to set up manually. See the example below.

§ Example

A session-log entry

Per §12 of CLAUDE.md, every project keeps a running log of substantive turns as a Claude-native artifact, mirrored to session-log.html. New entries are added newest-first. Here is a template entry, in that format, covering how to open a Claude Code session against a project once CLAUDE.md is in place.

Entry — template session-log.html
Question

Once a project has a CLAUDE.md in place, how do I actually start a Claude Code session against it?

Answer
VS Code extension. Open the project folder in VS Code, then either install "Claude Code" from the Extensions Marketplace directly, or run claude in VS Code's integrated terminal — it offers to install the companion extension automatically. Once installed, the extension reads CLAUDE.md from the project root as context for every session opened against that folder; no separate loading step is needed. Sessions are opened and driven from the extension's panel and the Command Palette rather than a specific fixed keybinding, since that detail varies by extension version — search the palette for "Claude Code" if the panel isn't already visible.
JetBrains plugin. The same "Claude Code" plugin is published to the JetBrains Marketplace for IntelliJ-family IDEs, with the same CLAUDE.md-on-open behavior.
CLI. Run claude from a terminal inside the project directory. This is editor-independent and is what the IDE extensions wrap under the hood.
Desktop or web app. The desktop app (Mac/Windows) and the web app at claude.ai/code cover the same workflow for projects pushed to a connected GitHub repository, for cases where a local editor isn't the point of entry.
